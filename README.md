# План презентации

- Название: "Простая архитектура: разработка и тестирование приложений по Маслову"

1. Проблематика. Список болей:
    - проблемы в архитектуре
        - примеры архитектур переусложнены паттернами, желанием сделать всё "универсально" и "на будущее"
        - запросы в базу непонятно как собираются, непонятно как оптимизировать
        - дилемма - база протекает в БЛ, или наоборот
        - медиатр и автомаппер!!!
    - проблемы с обработкой ошибок через исключения
        - забываем добавить обработчики
        - ошибки не отражены в сигнатуре метода
    - проблемы с тестированием
        - если тесты есть, то часто они бесполезные
        - проходят на окружении, которое непонятно как разворачивается
        - мокают то, что невозможно замокать (IQuerable), или мокают не так (InMemoryDb вместо настоящей базы)

2. Основная часть
    - рассказать про чистую архитектуру в eShopOnWeb
    - простая архитектура
        - чистое ядро без зависимостей
            - интерфейсы
            - неизменяемые доменные сущности
        - изменение доменных сущностей только через БД
        - установка границ через защитные интерфейсы: для каждой внешней зависимости свой слой
        - нет лишних абстракций над абстракциями (спецификация)
        - простор для возможных оптимизаций
        - демо
    - обработка ошибок
        - проблемы исключений
            - не отражены в сигнатуре
            - goto с контекстом
        - размеченные определения
            - определение, пример из F# / TS
            - в сигнатуре метода явно описаны все варианты 
            - нужно явно обработать все части размеченного определения
            - добавление нового варианта – ошибка времени компиляции
        - рассказ про мою либу
            - существующие варианты
            - сорц генератор
        - демо (баг с отсутствующей проверкой ParentId)
        - недостатки размеченных определений
            - зараженность методов (как с тасками)
            - повышается вложенность
            - коллеги не понимают, зачем это нужно
    - тестирование
        - интеграционные тесты с поднятием реального окружения
        - запуск приложения в докере
        - плюсы            
            - примеры использования АПИ
            - можно легко разрабатывать по TDD
            - тест-кейсы для быстрого воспроизведения бага
        - минусы
            - тесты долго выполняются

3. Выводы:
    - явное всегда лучше неявного
        - не придумывать своих конвенций
        - не использовать ambient-контексты
    - делать проще
    - не делать на будущее, заранее не делать универсально
    - бизнес-требования должны соответствовать коду

# Как накатить новую миграцию

В одной консоли:

- cd ArchitectureDemo.IntegrationTests
- docker-compose up

В другой консоли:

- cd ArchitectureDemo.DAL
- dotnet ef migrations add MigrationName

После работы в первой консоли:

- docker compose down -v
